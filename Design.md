# PaperBrain

PaperBrain is a Postgres + pgvector CLI/MCP project, which serves as a scientific question-centric local knowledge base and discovery engine for researchers. PaperBrain can works with AI agent, MCP client, and CLI.

It focuses on:
1. Paper import from local PDF files(based on docling)
2. Hybrid search (keyword + vector)
3. AI summaries(card generation) for papers, people, and topics
4. Data quality checks and auto-fixes
5. Exporting structured markdown files for use in tools like Obsidian


## Core commands

```bash
paperbrain setup #a wizard to set up the database connection and configuration(~/.config/paperbrain.conf), prompting for necessary information like database URL and credentials, and testing the connection to ensure it's properly configured
paperbrain init --url postgresql://postgres:PaperBrainLocal2026ChangeMe@localhost:5432/paperbrain --force #initialize the database with the given connection URL, creating necessary tables and indexes. If --force is used, it will drop existing tables and re-create them, erasing all existing data in the database.
paperbrain ingest --force-all --recursive /path/to/papers #ingest papers from local directory or pdf files, extracting metadata and text content, skipping papers that have already been ingested unless --force-all is used.
paperbrain browse "keyword" --type [paper|person|topic|all] #search for papers, people, topics, all cards using keyword
paperbrain search "your search query" --top-k 10 --include-cards  #perform a hybrid keyword + vector search across full text of papers, returning top-k most relevant papers. If --include-cards is used, also return relevant paper cards, person cards, and topic cards based on the search results.
paperbrain summarize --force-all #generate paper profiles, person profiles, and topic profiles using LLMs when papers are added or updated, and update corresponding paper, person, and topic pages in the database when --force-all is used, it will re-generate summaries and profiles for all papers, people, and topics in the database regardless of whether they have been updated or not
paperbrain lint #run data quality checks and auto-fix common issues like whitespace, dead links, and missing metadata
paperbrain stats #view corpus statistics (number of papers, authors, topics, etc.)
paperbrain export #export paper profiles, person profiles, and topic profiles as markdown files(obsidian vault format with bidirectional links)
```


## The design of table in database
### Paper Cards
PaperBrain Cards are **structured, article-type-specific summaries** generated by LLM from paper content

**For research articles**, it includes:
1. **slug** — `papers/<first-author-last-name>-<short-title>-<journal>-<year>-<journal>-<short-hash>`
2. **type** — `article`
3. **title** — From paper metadata
4. **authors** — List of all authors from paper metadata
5. **corresponding_authors** — linked to person cards
6. **journal** — From paper metadata
7. **year** — From paper metadata
8. **Key question solved** — The core research question addressed by the paper
9. **Why this question is important** — Scientific impact and relevance
10. **How the paper solves this question** — Methodology, approach, experiments design
11. **key findings and flow** —  the logical flow of the paper's sections and experiments and bullet points for key results with figure references
12. **Limitations of the paper** — Scope constraints, assumptions, and future work

**For review articles**, the summary includes:
1. **slug** — `papers/<first-author-last-name>-<short-title>-<journal>-<year>-<journal>-<short-hash>`
2. **type** — `review`
3. **title** — From paper metadata
4. **authors** — List of all authors from paper metadata
5. **corresponding_authors** — linked to person cards
6. **journal** — From paper metadata
7. **year** — From paper metadata
8. **Key goal of the review** — The main research question or theme the review addresses, and the scope of the review (e.g., specific subfield, time period, types of studies included)
9. **Key unsolved questions** — Bullet points highlighting major open problems in the field identified by the review, if any
10. **Why these unsolved questions are important** — Scientific and practical importance for each major open problem identified by the review, if any 
11. **Why these unsolved questions are still unsolved** — State of art, barriers, gaps in knowledge, and challenges for each major open problem identified by the review

**Paper card creation:**
- Generated by LLM from paper content and metadata
- Stored as `type: paper` pages in your corpus with slug `papers/<first-author-last-name>-<short-title>-<journal>-<year>-<journal>-<short-hash>`
- Updated whenever you run `paperbrain summarize` or add new papers
- Linked to corresponding author person cards and related topic cards

**Example Paper card frontmatter:**
```yaml
---
slug: papers/Chen-Nature-2024-abc123
type: research
title: "P53 Mutations and Cancer Progression"
authors: ["Stephen Chen", "Alice Research", "Bob Scientist"]
corresponding_authors: ["Alice Research <alice@university.org>", "Bob Scientist <bob@university.org>"]
journal: "Nature"
year: 2024
---
<!-- paperbrain_paper_summary:start -->
--> Key question solved: How do specific P53 mutations drive cancer progression?
--> Why this question is important: P53 is the most commonly mutated gene in cancer, and understanding its role is critical for developing targeted therapies.
--> How the paper solves this question: The authors use a combination of CRISPR gene editing, single-cell RNA sequencing from cohort studies with 1000 samples, and in vivo mouse models to analyze the effects of different P53 mutations on tumor growth and metastasis.
--> Key findings and flow:
- The paper starts with an introduction to P53 and its role in cancer, followed by a description of the experimental design.
- The first set of experiments uses CRISPR to introduce specific P53 mutations into cell lines, showing that certain mutations lead to increased cell proliferation and resistance to apoptosis (Figure 1).
- Next, single-cell RNA sequencing of patient tumor samples reveals distinct gene expression profiles associated with different P53 mutations, suggesting that they may drive cancer progression through different pathways (Figure 2).
- Finally, in vivo mouse models demonstrate that tumors with specific P53 mutations grow faster and are more likely to metastasize compared to those with wild-type P53, confirming the functional impact of these mutations (Figure 3).
--> Limitations of the paper: The study focuses on a limited set of P53 mutations and may not capture the full diversity of P53-driven cancers. Further research is needed to validate these findings in human clinical samples and across different cancer types.
<!-- paperbrain_paper_summary:end -->
```

### Person Cards

Person cards aggregate researcher profiles based on papers in your corpus.

**A person card includes:**
- **Person page slug** - `people/<normalized-email>` for unique identification
- **Name, email, affiliation** — From paper `corresponding_authors` metadata
- **Focus area** — linked to topic pages, extracted from their papers and grouped by themes
- **Big questions** — Top long-horizon research questions this researcher is focused on, extracted from their papers and grouped by themes. Each big question includes:
  - **Question** — The specific long-horizon research question
  - **Why important** — Strategic significance of that question
  - **Related papers** — Related paper slugs for that group (uncapped), also including papers where this researcher is first author or co-first author. linked bidirectionally from person page to paper pages and vice versa.

**Person card creation:**
- Extracted from `corresponding_authors` field in paper frontmatter
- A unique person card slug is generated from researcher email (`people/<normalized-email>`)
- LLM analyzes their papers to extract focus areas and research themes
- Person pages are created or updated as a `type: person` page in your corpus
- Person cards appear automatically in summarize output as `## Corresponding Author Cards`
- Links are maintained bidirectionally: papers link to researchers, researchers link to their papers

**Example person card frontmatter:**
```yaml
---
slug: people/alice-research
type: person
name: Alice Research
email: alice@university.org
affiliation: AI Lab, University
---

<!-- paperbrain_author_profile:start -->
- Focus area: Attention mechanisms in large language models
- Big questions:
  - Question: Can we build attention patterns that scale sub-linearly with context length?
    - Why important: Enabling models to process documents of arbitrary length
    - Related papers: papers/efficient-attention, papers/long-context-attention
  - Question: How can attention stay interpretable at frontier model scale?
    - Why important: Improves trust, debugging, and safety validation for production systems
    - Related papers: papers/attention-interpretability, papers/mechanistic-attention-analysis
<!-- paperbrain_author_profile:end -->
```

### Topic Cards

Topic cards is extracted from all big questions in all person cards and grouped by themes
**A topic card includes:**
- **Topic page slug** — `topics/<normalized-topic>`, LLM-extracted and normalized from big questions across person cards
- **Topic name** — Human-friendly name for the topic, extracted from big questions across person cards
- **Related big questions** — List of big questions from person cards that are grouped under this topic, each including:
  - **Question** — The specific long-horizon research question
  - **Why important** — Strategic significance of that question
  - **Related papers** — Related paper slugs for that group, linked bidirectionally from topic page to paper pages and vice versa.
  - **Related people** — Related person slugs for that group), linked bidirectionally from topic page to person pages and vice versa.

**Topic card persistence and linking:**
- Persisted as `type: topic` pages (`topics/<normalized-topic>`)
- Linked bidirectionally to paper and person pages
  - paper page `<->` topic page
  - person page `<->` topic page

**Example topic card frontmatter + profile block:**

```yaml
---
slug: topics/graph-neural-networks
type: topic
topic: Graph Neural Networks
---
<!-- paperbrain_topic_profile:start -->
## PaperBrain Topic Profile

- Topic: Graph Neural Networks
- Topic page: topics/graph-neural-networks
- Related big questions:
    - Question: How do we scale GNNs?
      - Why important: Scale enables real-world graph reasoning.
      - Related papers: papers/gnn-1, papers/gnn-2
      - Related people: people/ada-lovelace, people/grace-hopper
    - Question: How do we make GNNs more expressive?
      - Why important: Expressivity is key for modeling complex graph structures.
      - Related papers: papers/gnn-expressivity-1, papers/gnn-expressivity-2
      - Related people: people/alan-turing, people/ada-lovelace
<!-- paperbrain_topic_profile:end -->
```

## Skills
- setup(database initialization and configuration)
- ingest(ingest papers from local PDF files, extracting metadata and text content, skipping already ingested papers unless --force-all is used)
- summarize (generate paper summaries, person profiles, and topic profiles using LLMs when papers are added or updated, and update corresponding paper, person, and topic pages in the database when --force-all is used, it will re-generate summaries and profiles for all papers, people, and topics in the database regardless of whether they have been updated or not)
- lint (run data quality checks and auto-fix common issues like whitespace, dead links, and missing metadata)
- stats (view corpus statistics like number of papers, authors, topics, etc)
- browse (keyword search across paper/person/topic cards)
- search (hybrid keyword + vector search across paper full text, returning relevant papers and cards)
- export and organize (export paper/person/topic profiles as markdown files in obsidian vault format with bidirectional links, organized into folders by type)
