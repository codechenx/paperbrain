# LLM-Generated Person/Topic Cards Design

## Context and problem

PaperBrain currently derives person/topic card content with heuristic parsing and theme inference. The required behavior is:

1. Build person identities and paper links from paper cards.
2. Generate person `big_questions` from linked papers using LLM.
3. Generate topic cards using LLM from all person-card big questions.
4. Derive person `focus_area` after topic generation by linking each person to generated topic cards.
5. Retry once when LLM output is malformed or incomplete; if retry still fails, fail summarize with a clear error.

This design removes heuristic content synthesis for person/topic card generation while preserving deterministic identity/link integrity.

## Goals

1. Person card `big_questions` are generated by LLM from each person’s linked paper cards.
2. Person card `focus_area` is populated after topic generation from linked topic names/slugs.
3. Topic cards are generated by LLM from all big questions across all person cards.
4. Validation is strict and explicit: no silent fallback synthesis when LLM output is invalid.
5. Existing markdown/export structure from `Design.md` remains unchanged.

## Non-goals

1. Changing paper-card summary generation logic.
2. Changing export format/layout beyond maintaining existing compatibility.
3. Introducing a new persistence model or schema.

## Architecture overview

### Stage 1: deterministic person identity + linking

Use paper cards as source of truth for identity and relation seeds:

1. Parse `corresponding_authors` from each paper card.
2. Normalize identity (`name`, `email`, `affiliation`, `slug`).
3. Link all related paper slugs to each person seed.

This stage is deterministic and does not synthesize research content.

### Stage 2: LLM person-question generation

For each person seed:

1. Build an evidence bundle from all linked paper cards:
   - `slug`, `title`, `authors`, `journal`, `year`, `summary`, `corresponding_authors`.
2. Prompt LLM to generate strict JSON fields:
   - `big_questions` (non-empty array of objects with `question`, `why_important`, `related_papers`).
3. Merge with deterministic identity fields and linked papers.

### Stage 3: LLM topic generation

After all person cards are valid:

1. Build topic-generation input from all person cards’ `big_questions` plus known person/paper slugs.
2. Prompt LLM to return strict JSON topic cards:
   - `slug`, `type`, `topic`,
    - `related_big_questions` (`question`, `why_important`, `related_papers`, `related_people`),
    - `related_people`, `related_papers`.

### Stage 4: person focus-area linking from topics

After topic cards are valid:

1. For each person slug, collect all topic cards where the person appears in `related_people`.
2. Populate person `focus_area` from linked topic names (and keep topic-page linkage consistency).
3. Enforce non-empty `focus_area`; if no topic link exists for a person, fail with a clear validation error.

## Data contracts

### Person card contract (post-validation shape)

```json
{
  "slug": "people/<normalized-email-or-identity>",
  "type": "person",
  "name": "string",
  "email": "string",
  "affiliation": "string",
  "focus_area": ["string topic label derived from linked topics", "..."],
  "big_questions": [
    {
      "question": "string",
      "why_important": "string",
      "related_papers": ["papers/<slug>", "..."]
    }
  ],
  "related_papers": ["papers/<slug>", "..."]
}
```

Validation rules:

1. `big_questions` must be non-empty.
2. Every `big_questions[].related_papers` must be a subset of the person card’s linked papers.
3. Unknown or empty paper slugs are rejected.
4. `focus_area` is populated after topic generation and must be non-empty at final validation.

### Topic card contract (post-validation shape)

```json
{
  "slug": "topics/<normalized-topic>",
  "type": "topic",
  "topic": "string",
  "related_big_questions": [
    {
      "question": "string",
      "why_important": "string",
      "related_papers": ["papers/<slug>", "..."],
      "related_people": ["people/<slug>", "..."]
    }
  ],
  "related_people": ["people/<slug>", "..."],
  "related_papers": ["papers/<slug>", "..."]
}
```

Validation rules:

1. `related_people` references must exist in generated person cards.
2. `related_papers` references must exist in known paper slugs.
3. `related_big_questions` must be non-empty and internally consistent with top-level links.

## Error handling

1. If person-stage LLM output is malformed/invalid, retry the same stage once. If the retry is still invalid, stop summarize and raise a clear validation error.
2. If topic-stage LLM output is malformed/invalid, retry the same stage once. If the retry is still invalid, stop summarize and raise a clear validation error.
3. No regex/theme fallback content generation for person/topic cards.
4. Deterministic normalization (slug cleanup, dedupe, list normalization) is allowed only after schema-valid output.
5. If focus-area linking from validated topics yields an empty set for a person, stop summarize and raise a clear validation error.

## Testing strategy

Update adapter/service tests to cover:

1. Person `big_questions` are generated from LLM JSON content using linked-paper evidence.
2. Topic cards are generated from all person big questions via LLM JSON content.
3. Person `focus_area` is derived after topic generation from linked topics.
4. Invalid/malformed LLM JSON retries once, then fails summarize with explicit error.
5. Person/topic content no longer depends on heuristic theme inference functions.
6. Existing export compatibility remains intact for person/topic markdown structure.

## Impacted code areas

1. `paperbrain/adapters/llm.py`
   - Replace heuristic content derivation in person/topic generation paths with LLM-driven JSON generation + strict validators.
2. `paperbrain/services/summarize.py`
   - Keep two-pass orchestration (person then topic); ensure errors propagate clearly.
3. `tests/test_openai_adapter.py` and related summarize/export tests
   - Update/add regression coverage for new contracts and failure semantics.

## Rollout notes

1. Use existing `--force-all` replace semantics to refresh person/topic sets consistently.
2. Do not change existing CLI flags or user-facing command flow.
